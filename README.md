## เปิดใช้งาน

**ต้องมี Python 3.10+ จาก [python.org](https://www.python.org/downloads/)** (ตอนติดตั้งให้ติ๊ก

1. ดับเบิลคลิก **`run.bat`** → เบราว์เซอร์เปิดที่ `http://127.0.0.1:8000` เอง

> **ไม่มีโฟลเดอร์วิชาก็ใช้ได้** — เปิด `run.bat` แล้วลากรูปอะไรก็ได้มาวางบนหน้าเว็บเลย

หรือสั่งเอง:

รันเทสต์: `python -m pytest tests -q` (ต้องมีรูปใน `images/` ก่อน)

1. เลือกรูปจากแถบซ้าย (34 รูป มาจากโฟลเดอร์ของอาจารย์ทั้งหมด) หรือลากไฟล์ตัวเองมาวาง

## สิ่งที่ค้นพบตอนสร้าง — อ่านก่อนเข้าห้องสอบ

ระหว่างเทียบโค้ดอาจารย์กับสรุปของคุณทีละบรรทัด เจอจุดที่ **ขัดกันเอง** อยู่หลายจุด
ทุกจุดถูกตรึงไว้ด้วยเทสต์ใน `tests/` และขึ้นป้าย ⚠️ ในแท็บ "ข้อควรระวัง" ของเว็บ

### 1. สรุปของคุณ "ปัดเศษ" แต่โค้ดอาจารย์ "ตัดทิ้ง"

โค้ดอาจารย์ลงท้ายด้วย `.astype(np.uint8)` ซึ่ง**ตัดเศษทิ้ง** ส่วนสรุปคำนวณมือแบบ**ปัดเศษ**
ค่าที่ตกใกล้ .5 จะต่างกัน 1 หน่วย:

| กรณี | ค่าจริง | โค้ด (ตัดทิ้ง) | สรุป (ปัดเศษ) |
|---|---|---|---|
| Log `r=10, r_max=250` | 110.663 | **110** | **111** |
| Log `r=100, r_max=250` | 212.988 | **212** | **213** |
| CMYK ของ RGB(100,150,200) — C | 127.5 | **127** | **128** |
| CMYK ของ RGB(100,150,200) — M | 63.75 | **63** | **64** |
| Gamma `γ=0.5, r=100` | 161.276 | 161 | 161 ✓ ตรงกัน |

Gamma บังเอิญตรงกันเพราะค่าไม่ตกใกล้ .5 — เลยไม่มีใครสังเกต

**ตอบข้อสอบยังไง:** ข้อให้**คำนวณมือ** → ปัดเศษตามสรุป | ข้อให้**เขียนโปรแกรม** → โค้ดจะได้ค่าตัดทิ้ง
เว็บนี้ทำตามโค้ดอาจารย์ (ตัดทิ้ง) แต่ตารางในแท็บ "คำนวณทีละขั้น" โชว์ค่าดิบให้ปัดเองได้

### 2. สไลด์ใช้ `>` กับ `>=` ไม่ตรงกัน

| ไฟล์ | ตอนเอา T ไปแบ่งภาพ |
|---|---|
| `slide Aj/Otsu_Chapter`, `Intermean_Chapter`, `Thresholding_Chapter` | `out[img >= T] = 255` |
| `week7 real/Otsu_adaptation`, `Intermean_adaptation`, `InclassIntermeanLoop` | `out[img > T] = 255` |

ต่างกันแค่พิกเซลที่**เท่ากับ T พอดี** — เว็บนี้ยึด **`>=`** ตาม `slide Aj` (แหล่งหลัก) และตรงกับสรุปของคุณที่เขียน `r ≥ T`

### 3. `matrix_Rotatef` เขียนเครื่องหมาย sin ตรงข้ามกัน 2 ไฟล์

| ไฟล์ | เมทริกซ์ |
|---|---|
| `image_spatial_transformation.ipynb` (ภาพเทา) | `R[0,1] = +sin`, `R[1,0] = −sin` |
| `image_spatial_transformation_color.ipynb` (ภาพสี) | `R[0,1] = −sin`, `R[1,0] = +sin` |

หมุนคนละทิศกันเลย เว็บนี้ยึดเวอร์ชันภาพเทา **ถ้าข้อสอบให้คำนวณมือ ต้องดูให้ดีว่าโจทย์อ้างอิงอันไหน**

### 4. บั๊กในสไลด์ที่ต้องรู้

| จุด | ปัญหา | เว็บนี้ทำยังไง |
|---|---|---|
| `MedianBlur(img, k)` | รับ `k` แต่ข้างในเรียก `cv2.medianBlur(img, 3)` — `k` ไม่มีผล | ส่ง `k` จริง |
| `filterImage(img, k)` | `mask = np.ones([k,k])/9` หาร 9 ตายตัว ผิดเมื่อ k≠3 | หาร `k**2` (ตรงกับ `filterCustom` ที่อาจารย์เขียนถูกอยู่แล้ว) |
| `rgb_to_hsv` | `return int(h), int(s), int(v)` — S,V อยู่ [0,1] ถูกตัดเหลือ 0 หมด | เก็บเป็น float คูณ 255 ตอนแสดงผล — และ `int(h)` ยังทำให้ mask ใน cell 12 จริงๆ เป็น H ∈ [0, **41**) ไม่ใช่ [0, 40] |
| `mask = (hsv[:,:,0] >= 0) & (... <= 40)` (cell 12) | **ไม่มีตัวกรอง S/V** → พิกเซล Δ=0 (ดำ/เทา/ขาว) ได้ H=0 ถูกนับเป็น "สีแดง" หมด และ **เขียนสีแดงจริงไม่ได้** เพราะแดงคร่อม 0° (340–360 ∪ 0–20) แต่สูตรเป็นช่วงเดียว | **เก็บกับดักไว้ทั้งคู่** ใน `hue_mask` เพราะเป็นคำตอบข้อสอบ — ฉบับที่แก้แล้วอยู่ใน `color_select` (มี wrap + S/V floor + เลือกหลายสี) |
| `flip_image` (เวอร์ชันสี) | opt=0 ใช้ backward map แต่ opt=1,2 ใช้ forward map | ผลลัพธ์เหมือนกัน แต่ขึ้นป้ายเตือนวิธีคิด |
| `img_transform` | forward map + `int()` → ภาพมีรูโหว่ | **ไม่ใช่บั๊ก** อาจารย์ตั้งใจสอน แล้วแก้ด้วย `medianBlur` ทีหลัง — ลองต่อ Median ในโหมด pipeline ดู |

### 4.5 จุดที่ OpenCV กับ numpy หลอกเรา (เจอตอนเพื่อน clone ไปรัน)

ไม่ใช่เรื่องในสไลด์ แต่เป็นกับดักจริงที่ทำให้โค้ด "ดูเหมือนทำงาน" แล้วพังเงียบๆ:

| กับดัก | เกิดอะไรขึ้น |
|---|---|
| **`cv2.imread` / `cv2.imwrite` กับ path ภาษาไทย** | บน Windows `imread` คืน `None` และ `imwrite` คืน `False` **โดยไม่ raise** — clone ลงโฟลเดอร์ชื่อ `โปรแกรมจากเจ๋ง` แล้วอ่านรูปไม่ได้สักไฟล์ ทางแก้: `cv2.imdecode(np.fromfile(path, np.uint8))` และ `cv2.imencode(...)[1].tofile(path)` (ดู `backend/dip/common.py`) |
| **`1 + r` เมื่อ `r` เป็น uint8** | `1 + np.uint8(255)` = **0** (วนกลับ) → `np.log(0)` = `-inf` → **ภาพออกมาดำทั้งรูป** ต้อง `img.astype(np.float64)` ก่อนเสมอ |
| **`np.log()` ของ uint8** | คืน **float16** ไม่ใช่ float64 → `c` คลาดเคลื่อน (46.34 แทน 46.3187) |
| **`np.uint8(274)`** | numpy 1.x แค่เตือน แต่ **numpy 2.x โยน `OverflowError`** — `pip install numpy` วันนี้ได้ 2.x ต้อง `np.clip(x, 0, 255)` ก่อนแปลงเสมอ |
| **กำหนดค่า float ลง array uint8** | `out[y,x] = 234.7` → ได้ 234 (ตัดทิ้ง ไม่ใช่ปัดเศษ) ตรงกับ `.astype(uint8)` พอดี |
| **`S = Δ/Cmax` ระเบิดบนพิกเซลมืด** | `RGB(3,9,1)` — แทบดำสนิท — ได้ **S = 0.89** คือ "เขียวสดเต็มที่" ตามสูตร วัดจริงกับนอยส์เกือบดำ (0–13): **13% ตกในช่วงสีส้ม 20–60° และมี S เฉลี่ย 0.69** → **S floor กรองออกได้แค่ 6% แต่ V floor กรองออก 100%** สวนสัญชาตญาณ: จะกันพื้นหลังดำต้องใช้ **V** ไม่ใช่ S (แต่กันพิกเซลเทา/ขาวโพลนต้องใช้ **S** เพราะเทา 128 มี V=0.5) → ต้องมีทั้งคู่ |
| **`cv2.cvtColor(float32, COLOR_HSV2BGR)`** | **for-loop เขียนตามให้ตรงเป๊ะไม่ได้เลย** — cv2 ใช้ตาราง sector + `cvRound` ภายในของมันเอง วัดแล้วเพี้ยน **±1 บน ~27% ของพิกเซล** ทุก variant (float32/float64 × ปัด/ตัดทิ้ง) ถ้าเอามาเป็น engine หลักการ `loop == backend` ของทั้ง repo จะแดงถาวร → `bgr_from_hsv` เลยเขียนเองด้วย float64 + `np.rint` ผลพลอยได้: **แปลงไป-กลับได้ค่าเดิมเป๊ะทุกสี** (ทดสอบครบ 393,216 สี) ส่วนของ cv2 เพี้ยน 1 |
| **`np.rint` / `round()` ปัดครึ่งไป "เลขคู่" ไม่ใช่ "ขึ้น"** | `round(0.5) = 0` แต่ `round(1.5) = 2` — และ `threshold.float2int` ของโปรเจกต์นี้ปัดครึ่ง**ขึ้น** **ห้ามผสมกัน** ใน `bgr_from_hsv` จะพัง vec==loop ที่ `.5` พอดี |

`tests/test_unicode_paths.py` กับ `tests/test_all_functions_all_images.py` ตรึงทุกข้อนี้ไว้
และเทสต์ทั้งชุดรันผ่านทั้ง numpy 1.26 + cv2 4.12 และ numpy 2.4.6 + cv2 5.0

### 4.6 เลือกสี / ดรอปสีอื่น — `hue_mask` vs `color_select`

**อาจารย์ไม่เคยใช้ `cv2.cvtColor(BGR2HSV)` เลยสักที่** — ทั้ง `slide Aj` และ `week1`–`week11`
คำนวณ H เองด้วยสูตรมือเป็น**องศา 0–360** ตลอด (ทางลัด cv2 เก็บ H เป็น [0,179] คือปัดเป็นจำนวนเต็ม
แล้วหาร 2 → เทียบได้แค่องศาคู่ → **ไม่ตรงกับสูตรมือที่ขอบช่วง**: วัดแล้วต่างกัน 91/4096 พิกเซลบน lena)

| | `hue_mask` (★3) | `color_select` |
|---|---|---|
| ที่มา | `HSV_ColorSpace_new` cell 12 **เป๊ะๆ** | ไฟนอลข้อ 2 ของคุณเอง |
| ใช้ตอน | **ตอบข้อสอบ** | ใช้งานจริง |
| สีแดง (คร่อม 0°) | ❌ ได้ภาพดำสนิท | ✅ `(H≥low) \| (H≤high)` |
| พื้นหลังดำ/เทา | ❌ ถูกนับเป็นสีแดง | ✅ กรองด้วย S/V floor |
| หลายสีพร้อมกัน | ❌ | ✅ |

**`hue_mask` จงใจไม่แก้** — `test_hue_mask_keeps_the_professors_trap_on_purpose` ตรึงกับดักไว้กันคนมา "แก้" ทีหลัง

จุดที่ S/V floor เห็นผลชัด **ขึ้นกับโหมด drop** (วัดแล้ว ตรึงใน `test_the_sv_gate_is_visible_with_drop_white_but_not_drop_gray`):
- **`drop=ขาว` → เห็นชัดมาก**: เลือกสีแดงแล้วปิด gate → พิกเซลดำ (H=0) รอดมาเป็นจุดดำเต็มพื้นขาว
- **`drop=เทา` → ไม่เห็นเลย**: พิกเซลดำที่ "เก็บไว้" กับที่ "แปลงเป็นเทา" ก็ดำเหมือนกัน — แต่ความเสียหายไปโผล่ตอนต่อ `color_adjust` ที่ `V× > 1` แล้วนอยส์ถูกขยายขึ้นมา (วัดได้: ความสว่างพื้นหลัง 7.2 → 20.5)

### 5. น้ำหนัก grayscale รวมกันได้ 0.9999 ไม่ใช่ 1

`0.2989 + 0.5870 + 0.1140 = 0.9999` (ค่ามาตรฐาน NTSC คือ `0.299/0.587/0.114` ซึ่งรวมได้ 1 พอดี)
ต่างกันน้อยกว่า 1 ระดับสี ไม่มีผลกับภาพ แต่**ตอนตอบข้อสอบให้เขียนตามอาจารย์**

### 6. `edge_operator_meth` จัดการขอบภาพต่างจาก mean/median

| ฟังก์ชัน | ขอบภาพ |
|---|---|
| `filterCustom` (mean), `MedianCustom` | `out = img.copy()` → ขอบเก็บภาพเดิมไว้ |
| `edge_operator_meth` | `out = np.zeros_like(...)` → **ขอบเป็นสีดำ** |

จุดนี้ทำให้เทสต์ `test_code_matches_backend.py` จับได้ตอนสร้างเว็บ

### ทำไม backend รันแบบ vectorized แต่โชว์โค้ด for-loop

for-loop ทีละพิกเซลบนภาพ 512×512 ใช้เวลาหลายวินาที — ลากสไลเดอร์ไม่ไหว
